## Подход к поставке

Проект развивается вертикальными продуктовыми инкрементами. Базовый фотобанк создаётся до
валидации распознавания, чтобы эксперименты выполнялись на реальных загруженных оригиналах и
опирались на проверенный контур событий, медиатеки и публикации. Каждый продуктовый этап
завершается работающим пользовательским или операторским сценарием, пригодным для отдельной
поставки.

Целевая MVP-цепочка:

`событие → загрузка фото → обработка → публикация → поиск → корзина → оплата → защищённое скачивание`

Решения по storage, очередям, ML, векторному поиску, аутентификации, платежам,
наблюдаемости и биометрии сначала фиксируются отдельными ADR. Базовая архитектура остаётся
Django modular monolith + PostgreSQL + Docker Compose на Yandex Cloud VM.

## Этап 0. Надёжная инженерная основа

**Результат:** команда может безопасно разрабатывать и автоматически поставлять следующие
инкременты.

### Контекст будущего плана

- Проверить локальный PostgreSQL, Docker Compose, конфигурацию окружений, CI и production deployment.
- Добавить минимальную эксплуатационную основу: health/readiness checks, структурированные логи и
  безопасный запуск миграций.
- Зафиксировать процедуры отката и резервного копирования существующей базы.
- Не развивать предварительные `Event` и `Photo` как окончательную модель.
- Устранить дублирование активного интерфейса и прототипа либо явно обозначить `src/proto` как
  архивный источник.
- Обеспечить одинаковую конфигурационную модель локально, в CI и production.

### Контекст автоматизированных тестов

- CI поднимает PostgreSQL и выполняет все обязательные проверки репозитория.
- Django system check и migration drift check проходят на чистой конфигурации.
- Smoke-тест проверяет запуск приложения и health endpoint.
- Контейнерный тест проверяет старт web-приложения с PostgreSQL.
- Проверяется отказ запуска при отсутствии обязательных секретов.
- Deployment workflow проверяется статически без публикации production-секретов.

## Этап 1. Базовый каталог событий

**Результат:** администратор создаёт и публикует событие через Django Admin, а посетитель видит
опубликованные события в публичном каталоге.

### Контекст будущего плана

- Спроектировать минимальную целевую модель события: название, даты, город, описание, обложка,
  тип `free` или `paid` и состояние публикации.
- Использовать Django Admin как единственный интерфейс создания и редактирования событий.
- Выполнить безопасный переход от предварительной модели `Event`.
- Добавить публичные главную каталога и страницу события с понятными пустыми состояниями.
- Показывать публично только опубликованные события.
- Не включать location, назначения фотографов, аудит и отдельный кабинет оператора до появления
  подтверждённой продуктовой потребности.

### Контекст автоматизированных тестов

- Модельные тесты ограничений дат, slug, типа и состояния публикации.
- Проверка создания и публикации события через административный контур.
- Неопубликованное событие отсутствует в каталоге и недоступно по прямому URL.
- UI smoke-тесты каталога, страницы события, пустого каталога и отсутствующего события.
- Migration-тесты сохранности существующих прототипных данных.

## Этап 2. Доступ фотографов и простая загрузка

**Результат:** авторизованный фотограф выбирает любое событие и загружает в него пакет фотографий,
а оригиналы остаются приватными.

**Статус:** подробное проектирование завершено, продуктовая реализация этапа не начата.

- [Подробный план реализации этапа 2](2026-07-13-stage-2-photographer-upload.md)
- [Согласованный design загрузки](../superpowers/specs/2026-07-13-stage-2-photographer-upload-design.md)

### Уже готово для этапа

- [x] Django users, session authentication и Django Admin для управления пользователями.
- [x] Каталог событий с типом `free`/`paid`, состоянием публикации и административным контуром.
- [x] ADR 0006 о раздельных public/private storage boundaries и неизменяемых object keys.
- [x] Базовая S3-интеграция с Yandex Object Storage для публичных обложек событий.
- [x] PostgreSQL, Docker Compose, CI/CD, staging и базовый production design system.
- [x] Согласованный и независимо проверенный design браузерной загрузки этапа 2.

### Остаётся реализовать

- [ ] Принять ADR по роли фотографа, прямой приватной загрузке и request-driven ingestion.
- [ ] Настроить отдельный private bucket, IAM, CORS, секреты и feature flag.
- [ ] Добавить модели и безопасные миграции пакетов, файлов и приватных оригиналов.
- [ ] Реализовать Photographer permission, login/logout, ownership и Admin inspection.
- [ ] Реализовать presigned POST, ETag-bound проверку, promotion и stale cleanup.
- [ ] Реализовать JSON control endpoints и браузерную очередь из четырёх параллельных загрузок.
- [ ] Добавить permission, storage, migration, API, JavaScript, visual, contract и load tests.
- [ ] Пройти staging rollout; только после этого отметить этап и архитектуру реализованными.

### Зафиксированные границы реализации

- До реализации принять ADR по минимальной модели аутентификации и ролям.
- Использовать Django users и группу или permission `Photographer`; администраторы продолжают
  использовать Django Admin, а обычные посетители остаются анонимными.
- Не создавать назначения фотографов: на первом этапе каждый фотограф видит все события и может
  загружать в любое из них.
- Создать пакетную загрузку с обязательными событием и зафиксированным загрузившим пользователем.
- Хранить оригиналы приватно по неизменяемым storage keys, не раскрывая keys в HTML или API.
- Хранить состояния пакета и отдельных файлов, включая частичный сбой и возможность безопасного
  повтора без повторной загрузки успешных файлов.
- Для этого этапа использовать прямую загрузку отдельных JPEG из живой страницы в private Object
  Storage; не добавлять ZIP, desktop client, восстановление очереди, broker или worker.
- До реализации принять ADR по private ingestion, жизненному циклу файлов и request-driven
  выполнению; не менять принятую ADR 0006 без superseding ADR.

### Контекст автоматизированных тестов

- Permission matrix для администратора, фотографа и анонимного посетителя.
- Фотограф видит все события; посетитель не получает доступ к upload или Django Admin.
- Частичный сбой пакета сохраняет успешные загрузки и показывает ошибки отдельных файлов.
- Повтор не создаёт дубликаты записей или оригиналов.
- Storage contract tests с выбранным тестовым сервисом.
- Security-тест: оригинал и внутренние storage keys недоступны публично.

## Этап 3. Медиатека, публикация и производные

**Результат:** администратор публикует готовые фотографии; посетитель просматривает корректную
производную и может скачать оригинал бесплатной фотографии.

### Контекст будущего плана

- До реализации принять ADR по `free`/`paid` media access policy и контролируемой выдаче оригинала.
- Извлекать EXIF, корректировать ориентацию и формировать thumbnail и preview.
- Для `paid`-события создавать уменьшенный watermarked preview; для `free` — уменьшенную копию без
  watermark.
- Разделить processing state и publication state: успешная загрузка или обработка не публикует фото
  автоматически.
- Показывать на странице события только опубликованные и успешно обработанные фотографии.
- Для `paid`-фото до commerce показывать только watermarked preview и нейтральное сообщение о том,
  что покупка появится позже; публичное скачивание отсутствует.
- Для `free`-фото разрешить анонимное скачивание оригинала через контролируемую выдачу приложения
  или короткоживущий доступ без раскрытия постоянного storage key.
- Использовать имя файла и время загрузки только как явно отмеченный fallback при отсутствии EXIF.
- Не включать ML-распознавание в критерии готовности этапа.

### Контекст автоматизированных тестов

- Unit-тесты EXIF, ориентации и fallback-правил времени.
- Проверка разных производных и watermark policy для `free` и `paid`.
- Необработанное или неопубликованное фото отсутствует в публичной галерее.
- Free-оригинал доступен анонимно только через контролируемый download flow.
- Paid-оригинал недоступен, а HTML/API не раскрывают его URL или storage key.
- Интеграционный сценарий: upload → derivatives → явная публикация → галерея → free download.

## Этап 4. Повторяемая processing-платформа

**Результат:** разработчик повторно запускает версионированный анализ ранее загруженных фото с
новыми параметрами и сравнивает сохранённые результаты через CLI и файлы.

### Контекст будущего плана

- Реализовать recognition как логически изолированный worker/контейнер с явным контрактом задания и
  результата, сохранив Django владельцем продуктовых правил, jobs и transactional state.
- До выбора task framework, broker и retry semantics принять отдельный ADR.
- Хранить immutable processing runs: фото, analyzer и его версию, параметры, статус, время,
  диагностические данные и ссылку на версионированный результат.
- Каждый повторный запуск создаёт новый run и не изменяет результаты предыдущего.
- Обрабатывать приватный оригинал без повторной загрузки; все jobs и результаты ограничивать
  событием.
- Добавить минимум один реальный простой analyzer для содержательной проверки контракта; выбор bib
  и face моделей в этот этап не входит.
- Запускать и сравнивать результаты через CLI; экспортировать версионированные JSON/CSV-файлы без
  отдельного операторского dashboard.
- Не выделять independently deployed business service без измеренной необходимости и нового ADR.

### Контекст автоматизированных тестов

- Contract tests между Django и worker для успешного результата, ошибки, retry и несовместимой
  версии контракта.
- Повторный запуск с теми же или другими параметрами сохраняет отдельный immutable run.
- Старые результаты остаются доступными и сравнимыми после нового запуска.
- Worker не получает оригинал или metadata фотографии из другого события.
- CLI возвращает ненулевой код при полном или частичном сбое и формирует валидный JSON/CSV-отчёт.
- Интеграционный сценарий: существующий original → два runs → два сохранённых набора результатов.

## Этап 5. Техническая валидация поиска по стартовому номеру

**Результат:** принято обоснованное решение, позволяют ли реальные фотографии реализовать поиск по
стартовому номеру с приемлемым качеством и стоимостью.

### Контекст будущего плана

- Подготовить репрезентативный датасет из 500–2000 фотографий загруженных событий.
- Разметить эталонные номера, допустимые crop-зоны, сложные ракурсы и отрицательные примеры.
- Через processing runs сравнить кандидатов для bib detection и OCR.
- Измерить precision/recall, долю фото без результата, время, требования CPU/GPU и стоимость.
- Проверить коммерческие лицензии кода, моделей и весов.
- Оформить результаты как воспроизводимый benchmark и принять ADR по bib/OCR-компонентам.
- Если пороги качества не достигнуты, определить human-in-the-loop сценарий либо остановить функцию.

### Контекст автоматизированных тестов

- Версионированный fixture-набор с эталонной разметкой.
- Benchmark запускается фиксированными версиями моделей и параметров processing run.
- Метрики считаются автоматически отдельно для простых и сложных кадров.
- Проверяется строгая изоляция результатов по `event_id`.
- Регрессионный порог не позволяет незаметно ухудшить точность или производительность.
- Малый fixture-набор используется в integration pipeline, полный benchmark запускается вне CI.

## Этап 6. Поиск фотографий по стартовому номеру

**Результат:** пользователь вводит номер внутри события и получает вероятные фотографии,
отсортированные по времени; оператор может исправить результат.

### Контекст будущего плана

- Обрабатывать только crop-зоны допустимых типов: грудь, спина, шлем, велосипедная или рулевая
  табличка.
- Хранить несколько OCR-кандидатов с confidence, bbox, source и версией модели.
- Применять формат номера конкретного события и разрешённые постфильтры.
- Поддерживать несколько участников на одном фото.
- Ручное подтверждение, исправление или отклонение имеет приоритет над автоматикой.
- Поиск всегда ограничивается текущим событием.
- Другие события не смешиваются с основной выдачей и могут показываться только отдельным явно
  подписанным блоком.
- Ошибки ML остаются видимыми и повторяемыми из операторского интерфейса.

### Контекст автоматизированных тестов

- Unit-тесты нормализации OCR и форматов номеров события.
- Проверка нескольких кандидатов и нескольких номеров на фотографии.
- Тест приоритета manual correction над auto candidate.
- Negative tests против совпадений из другого события.
- Проверка сортировки по времени и отдельного блока других событий.
- Contract tests взаимодействия Django с выбранным ML-компонентом.
- Регрессионные тесты на размытие, наклон, перекрытие и ложные цифровые объекты.
- End-to-end сценарий: обработка → поиск → ручная коррекция → изменившаяся выдача.

## Этап 7. Модерация, жалобы и исключения из поиска

**Результат:** оператор управляет ошибочными и нежелательными публикациями без ручного удаления
исходных данных.

### Контекст будущего плана

- Скрывать фото целиком или исключать отдельный номер/лицо из индекса.
- Обрабатывать жалобы и запросы на удаление.
- Различать скрытие из публичной выдачи, исключение из поиска и физическое удаление.
- Все административные действия должны иметь автора, причину, время и audit trail.
- Повторная ML-индексация не должна отменять ручное решение.
- Определить retention и безопасное удаление производных файлов.

### Контекст автоматизированных тестов

- Скрытая фотография исчезает из каталога и всех типов поиска.
- Suppression сохраняется после повторной индексации.
- Проверяются права на модерацию и неизменяемость аудита.
- Тестируется последовательность complaint → review → hide/remove/restore.
- Проверяется удаление производных файлов без случайного удаления чужих объектов.
- Security-тесты запрещают доступ к скрытому объекту по известному URL.

## Этап 8. Правовая и техническая валидация поиска по лицу

**Результат:** правовые условия зафиксированы и benchmark показывает, можно ли безопасно и с
приемлемым качеством реализовать вероятностный поиск по лицу.

### Контекст будущего плана

- До обработки selfie или выпуска функции утвердить правовое основание, consent, retention,
  удаление, контроль доступа и incident handling для биометрических данных.
- Подготовить отдельный face dataset и через processing runs сравнить detection и embeddings.
- Измерить качество, ошибки, throughput, стоимость и лицензионные ограничения независимо от bib.
- Принять ADR по face model, vector storage и порогам перехода от PostgreSQL/pgvector к отдельной БД.
- Зафиксировать вероятностный UX, event scoping, suppression и удаление до продуктовой реализации.

### Контекст автоматизированных тестов

- Версионированный обезличенный fixture-набор и фиксированные версии моделей.
- Автоматический расчёт quality, throughput и failure metrics.
- Проверка строгой изоляции benchmark-результатов по `event_id`.
- Проверка удаления тестовых selfie и embeddings согласно утверждённому retention.
- Полный face benchmark остаётся отдельным от CI quality gate.

## Этап 9. Поиск фотографий по лицу

**Результат:** пользователь загружает selfie в рамках выбранного события и получает вероятные
совпадения без утверждения личности.

### Контекст будущего плана

- Реализовывать этап только после успешного завершения этапа 8 и принятия связанных ADR.
- Хранить bbox, embedding reference, confidence, model version и дату индексации.
- Использовать selfie только для конкретного запроса и события с утверждённым минимальным retention.
- Описывать результаты как вероятные совпадения и поддержать исключение лица и удаление embeddings.
- Не смешивать результаты других событий с основной выдачей.

### Контекст автоматизированных тестов

- Строгий фильтр `event_id` на уровне запроса и vector search.
- Contract tests создания, поиска и удаления embeddings.
- Тесты confidence thresholds и устойчивого порядка результатов.
- Проверка удаления временного selfie и истечения retention.
- Suppressed/hidden faces не возвращаются после переиндексации.
- UI-тесты вероятностной формулировки и согласия; security-тесты биометрических операций.

## Этап 10. Корзина, цены и промокоды

**Результат:** пользователь формирует корзину из опубликованных фотографий и получает
воспроизводимый итог заказа со скидкой.

### Контекст будущего плана

- Определить модель цены и лицензии до начала реализации этапа.
- Поддержать анонимную корзину либо учётную запись согласно ADR аутентификации.
- Зафиксировать цену товара в момент формирования заказа, не полагаясь на текущую цену фото.
- Промокоды: процентная или фиксированная скидка, срок действия, лимит использований и опциональная
  привязка к событию.
- Сервер является источником расчёта итоговой суммы.
- Не создавать download entitlement до подтверждённой оплаты.
- Исключить из покупки скрытые и недоступные фотографии.

### Контекст автоматизированных тестов

- Unit-тесты денежных расчётов с `Decimal` и правилами округления.
- Табличные тесты всех типов промокодов и граничных дат.
- Проверка лимитов использования и конкурентного применения последнего доступного кода.
- Защита от подмены цены клиентом.
- Скрытая или снятая с продажи фотография корректно блокирует checkout.
- Проверка изоляции пользовательских и сессионных корзин.
- End-to-end сценарий: галерея → корзина → промокод → подтверждённая сумма заказа.

## Этап 11. Реальная оплата, заказы и защищённое скачивание

**Результат:** пользователь оплачивает заказ через реального провайдера и получает доступ только к
купленным файлам.

### Контекст будущего плана

- До реализации принять ADR по платёжному провайдеру, webhook-контракту, возвратам и download policy.
- Создавать заказ с неизменяемым составом и суммой до перехода к оплате.
- Аутентифицировать webhook и обрабатывать его идемпотентно.
- Только подтверждённая оплата создаёт entitlement на конкретные order items.
- Генерировать purchased export отдельно от публичного preview.
- Выдавать короткоживущую signed URL либо авторизованный streaming response.
- Аудировать скачивания, возвраты и отозванные доступы.
- Оригиналы не становятся публичными ни при одном статусе заказа.

### Контекст автоматизированных тестов

- Contract tests с sandbox API выбранного провайдера.
- Проверка подписи webhook, повторной доставки, задержанного и неверного callback.
- State-machine тесты заказа: created, pending, paid, failed, refunded.
- Конкурентные callbacks не создают дублирующие оплаты или entitlement.
- Нельзя скачать неоплаченную, чужую, возвращённую или не входящую в заказ фотографию.
- Signed URL имеет короткий TTL и ограничен нужным объектом.
- End-to-end sandbox-сценарий: checkout → оплата → webhook → заказ → скачивание.
- Отдельные тесты восстановления после временной недоступности провайдера.

## Этап 12. Эксплуатационная готовность MVP

**Результат:** весь MVP измеримо и восстанавливаемо работает в production.

### Контекст будущего плана

- Принять ADR по observability, backup targets, retention, RPO/RTO и HTTPS/media delivery.
- Добавить метрики web-приложения, очередей, длительности обработки, ошибок ML, платежей и выдачи
  файлов.
- Настроить alerting, capacity limits и runbooks.
- Провести проверяемое восстановление PostgreSQL и media metadata.
- Определить безопасное повторное развёртывание и rollback.
- Выполнить нагрузочные проверки ожидаемого размера события.
- Подтвердить юридические документы, лицензии моделей и правила биометрической обработки.

### Контекст автоматизированных тестов

- Synthetic monitoring публичного каталога, поиска и checkout.
- Нагрузочные сценарии загрузки, индексирования, поиска и скачивания.
- Проверка alert conditions на остановку worker, рост очереди и ошибки платежей.
- Автоматизированная проверка создания backup; restore drill проводится по runbook.
- Smoke-тест после deployment и проверка обратной совместимости миграций.
- Security regression: приватность оригиналов, права операторов, webhook и биометрические данные.
- Полный end-to-end сценарий MVP на staging.

## Общие правила тестирования этапов

- Поведение разрабатывается через failing test → минимальная реализация → targeted tests → полный
  regression suite.
- Основные интеграционные тесты используют PostgreSQL, а не SQLite.
- Внешние сервисы покрываются двумя уровнями: быстрые детерминированные contract doubles в CI и
  sandbox-интеграция в отдельном pipeline.
- Для ML фиксируются версия модели, fixture dataset и пороги качества; случайные результаты не
  должны делать CI нестабильным.
- Каждый этап добавляет минимум один end-to-end happy path и проверки ключевых отказов.
- Обязательный regression gate: Ruff format, Ruff lint, mypy, pytest с coverage, Django check и
  migration drift check.
- Миграции PostgreSQL планируются отдельно и проверяются на блокировки, совместимость развёртывания
  и сохранность данных.
- Изоляция по событию, приватность оригиналов, приоритет ручных решений и entitlement после оплаты
  повторно проверяются на всех затрагивающих их этапах.

## Post-MVP

После подтверждения метрик MVP отдельными продуктовыми инициативами могут стать:

- базовый редактор и рамки события;
- поиск похожих фотографий;
- автоматический выбор лучших кадров;
- кабинеты партнёров и организаторов;
- интеграции с регистрационными платформами;
- мобильные приложения и публичный API.

Каждая инициатива требует отдельного discovery, оценки спроса и самостоятельного roadmap; она не
включается в критерии готовности MVP.

## Зафиксированные предположения

- Горизонт roadmap: подготовка, полный MVP и production readiness.
- Коммерческий этап использует реального платёжного провайдера, а не mock как конечный результат.
- Django modular monolith, PostgreSQL и текущая контейнерная модель сохраняются.
- Администраторы управляют событиями через Django Admin; пользователи группы `Photographer` видят
  все события и могут загружать в любое из них до появления необходимости в назначениях.
- Бесплатные события разрешают контролируемое анонимное скачивание оригинала; платные до commerce
  показывают только уменьшенный watermarked preview без скачивания.
- Recognition сначала реализуется как worker/контейнер с версионированным контрактом и immutable
  runs, а не как independently deployed business service.
- React/Next.js, Celery, broker, S3-провайдер, pgvector/Qdrant, ML-модели и платёжный провайдер не
  выбираются молча: для них требуются исследования и ADR.
- Поиск по лицу не выпускается до утверждения биометрической политики.
- Редактор, рекомендации, похожие кадры и партнёрские интеграции находятся за пределами MVP.
